## 贡献方式

你可以通过以下方式为项目做出贡献：

### 1. 内容贡献

- **修正错误**：发现技术错误、拼写错误、表述不清的地方
- **补充内容**：增加案例、代码示例、图表、参考文献
- **翻译**：帮助翻译成其他语言（英文、日文等）
- **优化排版**：改进Markdown格式、图表绘制

### 2. 代码贡献

- **修复Bug**：hsm-lite代码的问题修复
- **功能扩展**：增加新功能（如RSA/ECC支持、真实AES实现等）
- **性能优化**：提高代码效率、减少资源占用
- **测试完善**：增加测试用例、性能基准测试

### 3. 其他贡献

- **提交Issue**：反馈问题、提出建议
- **参与讨论**：在Issue中分享观点、回答问题
- **推广项目**：推荐给朋友、分享到社交媒体
- **提供案例**：分享你的实际应用经验

## 如何提交Issue

### Issue类型

我们欢迎以下类型的Issue：

- 🐛 **Bug报告**：代码错误或内容错误
- 💡 **功能建议**：新功能或改进建议
- 📚 **内容问题**：技术内容错误、表述不清
- ❓ **疑问求助**：学习过程中的问题
- 🎯 **应用案例**：分享你的实际应用经验

### Issue模板

提交Issue时，请包含以下信息：

```markdown
**类型**：[Bug报告/功能建议/内容问题/疑问求助]

**描述**：
简要描述问题或建议

**具体内容**：
- 问题所在章节/代码位置
- 具体的错误信息或现象
- 期望的正确结果或改进方案

**环境信息**（如果是代码问题）：
- 操作系统：[如 Ubuntu 20.04]
- GCC版本：[如 gcc 9.4]
- 其他相关信息

**截图/日志**（如果适用）：
粘贴相关的截图或日志输出
```

## 如何提交Pull Request

### 准备工作

1. **Fork项目**
   ```bash
   # 在GitHub上点击Fork
   git clone https://github.com/你的用户名/hsm-book.git
   cd hsm-book
   ```

2. **创建分支**
   ```bash
   git checkout -b feature/你的功能名称
   # 或
   git checkout -b fix/你的修复内容
   ```

### 内容修改

#### Markdown文件修改

1. 确保Markdown格式规范
2. 保持与现有内容风格一致（场景→问题→隐喻→技术→总结）
3. 使用正确的章节层级（##, ###）
4. 代码块使用正确的语法标注

#### 代码修改（hsm-lite）

1. 遵循现有代码风格
2. 添加必要的注释
3. 确保代码可编译
4. 测试修改后的功能

### 提交PR

1. **提交修改**
   ```bash
   git add .
   git commit -m "清晰的提交信息"
   git push origin feature/你的功能名称
   ```

2. **创建Pull Request**
   - 在GitHub上点击"New Pull Request"
   - 填写PR标题和描述
   - 关联相关的Issue（如有）

3. **PR描述模板**
   ```markdown
   ## 变更类型
   - [ ] 内容修正
   - [ ] 功能增加
   - [ ] Bug修复
   - [ ] 文档改进
   - [ ] 其他
   
   ## 变更描述
   简要说明本次修改的内容
   
   ## 关联Issue
   关闭 #[Issue编号]（如有）
   
   ## 测试说明
   - [ ] 已本地测试
   - [ ] 已验证Markdown渲染
   - [ ] 已编译并运行代码
   
   ## 检查清单
   - [ ] 代码风格一致
   - [ ] 注释完整清晰
   - [ ] 无编译警告
   - [ ] Markdown格式正确
   ```

## 内容规范

### 写作风格

本书采用统一的写作风格：

1. **场景引入**：每节开头用"一个场景：..."引入
2. **问题提出**：用"一个问题浮出水面：..."引导思考
3. **隐喻使用**：使用生动的隐喻（保险箱、普通话、身份证等）
4. **技术内容**：准确的技术描述和代码示例
5. **总结收尾**：用小结总结本节要点

示例：

```markdown
## 4.5 Slot管理模块：Token的"注册中心"

### 一个问题：Slot如何被管理？

假设你刚刚读完第三章，了解了PKCS#11的Slot概念。

你可能会困惑：软件HSM没有物理插槽，Slot是什么？

### "注册中心"的设计意图

SlotManager就像政府注册中心...

（技术内容）

### 小结

Slot管理是Token的"注册中心"，确保...
```

### Markdown规范

- **标题层级**：
  - 二级标题：章节名（如 ## 4.5）
  - 三级标题：子节（如 ### 一个问题）
  - 四级标题：细节（如 #### 示例代码）

- **代码块**：
  ```c      // C语言
  ```bash   // Shell命令
  ```makefile // Makefile
  ```

- **表格格式**：
  ```markdown
  | 列1 | 列2 |
  |:---|:---|
  | 内容 | 内容 |
  ```

### 技术内容规范

- **术语准确**：使用PKCS#11标准术语（Slot、Token、Session、Object）
- **代码注释**：函数、结构体必须有注释
- **图表说明**：ASCII图必须清晰可读

## 代码规范

### C代码规范（hsm-lite）

遵循Linux内核编码风格：

```c
/**
 * @brief 函数简要说明
 * @param param1 参数说明
 * @return 返回值说明
 */
CK_RV function_name(CK_SESSION_HANDLE hSession)
{
    CK_RV ret = CKR_OK;
    
    if (hSession >= HSM_MAX_SESSIONS) {
        return CKR_SESSION_HANDLE_INVALID;
    }
    
    return ret;
}
```

**要点**：
- 函数返回类型单独一行
- Tab缩进
- 注释使用Doxygen格式
- 返回值使用PKCS#11标准错误码

### Makefile规范

```makefile
CC = gcc
CFLAGS = -Wall -Wextra -O2 -std=gnu11

all: hsm_test

hsm_test: hsm_test.c hsm_lite.c
	$(CC) $(CFLAGS) -o $@ $^

clean:
	rm -f hsm_test *.o

.PHONY: all clean
```

## 审核流程

### 审核标准

我们审核PR时关注：

1. **内容准确性**：技术内容是否正确
2. **风格一致性**：是否与现有风格一致
3. **质量完整性**：是否有完整注释和说明
4. **测试验证**：代码是否可运行

### 审核时间

- 小修改：1-3天
- 大修改：3-7天
- 新功能：7-14天

### 审核反馈

审核完成后，会给出以下反馈：

- ✅ **通过**：PR被接受
- ⚠️ **需要修改**：会指出需要改进的地方
- ❌ **拒绝**：会说明原因
